# Learning: Path Truth Before Product Work

When using project orchestration commands, the most important invariant is that the working pointer matches the human's real development path. In this session, incubate initially followed ghq convention, but the user was actively developing in `~/Git/me/vibe-code/jit-flow`. Relinking `ψ/incubate/mahirocoko/jit-flow` to the local path immediately reduced ambiguity and ensured all future "work on jit-flow" actions target the intended repository.

I also confirmed that project understanding must come from implementation, not positioning docs alone. Combining commit history with route/service/schema inspection gave a truthful status map: core daily task and goal flows are wired, journey UX is highly developed, and several modules remain placeholder-level. This classification is essential for honest planning and prevents overestimating completion based on polished UI.

Practical rule: establish path truth first, then do architecture/status analysis, and report completion by wiring depth (`implemented`, `partial`, `placeholder`) rather than visual completeness.
