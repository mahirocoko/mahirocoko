# สรุปภาษาไทย (แบ่งตามช่วงเวลา)

## [00:00-09:59]
- เกริ่นว่าคนดูถามบ่อยว่า "จะเริ่มทำ Design System ยังไง"
- คลิปนี้เป็นตอนแรกของซีรีส์ เน้นวางรากฐานด้วย Tokens และ Variables ก่อนทำคอมโพเนนต์
- อธิบายภาพรวมสถาปัตยกรรมตัวแปร: brand, alias, mapped collections และแนวคิด responsive collection
- เป้าหมายปลายทางคือมีระบบคอมโพเนนต์ครบสำหรับพอร์ตหรือใช้งานจริงในองค์กร

## [10:00-19:59]
- ลงมือสร้าง color scales แบบเร็ว โดยกรอก/คัดลอกชุดค่า hex เป็นกลุ่มสีต่าง ๆ
- ย้ำให้จัด workflow ให้ทำซ้ำได้ง่าย (duplicate และเติมค่าเป็นแพตเทิร์น)
- เน้นว่าการตั้งค่าเริ่มต้นควร practical ก่อน แล้วค่อยปรับละเอียดภายหลัง

## [20:00-29:59]
- เริ่ม map สี neutral (gray scale) และเชื่อม black/white ให้สอดคล้องกับระบบหลัก
- วางกลุ่ม semantic เช่น warning โดยผูกค่ากลับไปยัง brand tokens
- แนวคิดหลักคือจัดชื่อและโฟลเดอร์ให้เป็นระบบ เพื่อให้ map ต่อในขั้นตอนถัดไปง่าย

## [30:00-39:59]
- สร้าง token ระดับการใช้งานจริง เช่น icon colors (default/info/success/warning/error)
- ต่อด้วย border tokens (primary/secondary/state) เพื่อรองรับสถานะ UI
- สื่อให้เห็นการไหลจาก "สีดิบ" ไปเป็น "semantic token" ที่คอมโพเนนต์ใช้งานได้ทันที

## [40:00-49:59]
- เข้าสู่ typography scale: อธิบายแนวคิด modular scale (เช่น major third)
- เริ่มจากฐาน 16px และปรับให้เข้ากับ grid ที่ใช้งานจริงใน Figma
- เน้นการตัดสินใจเชิงระบบ: เลือกกติกาที่ทีมใช้ร่วมกันได้ มากกว่าตามสูตรเป๊ะทุกค่า

## [50:00-59:59]
- ปรับ line-height ตามระดับ heading/body โดยยังยึดกริดอ่านง่าย
- ชี้ว่าการตั้งค่าขนาด/ระยะห่างไม่มีคำตอบเดียว ต้องบาลานซ์การอ่านและ accessibility
- แนวทางคือ "ตั้งค่าที่ใช้งานได้ก่อน" แล้ววนกลับมาจูนหลังเห็นการใช้งานจริง

## [60:00-69:59]
- สร้าง text styles เพิ่มเติม เช่นลิงก์ (underline) และน้ำหนักตัวอักษรหลายแบบ
- จัดกลุ่ม/ตั้งชื่อ style ให้ชัด เช่น body-small-link, body-medium-link
- โฟกัสที่ความสม่ำเสมอของ naming เพื่อให้ทีมค้นหาและใช้งานร่วมกันง่าย

## [70:00-79:59]
- อธิบายกรณี border width/radius ว่าบางทีมสามารถจบที่ alias ได้เลย
- ชี้ความต่างเชิงสถาปัตยกรรมระหว่าง alias กับ mapped ว่าบางครั้ง mapped อาจซ้ำ alias
- ย้ำหลักคิด: ทำ token ให้ generic พอสำหรับ reuse ไม่ผูกกับคอมโพเนนต์เฉพาะตัว

## [80:00-85:39]
- แตะเรื่อง responsive modes (เช่น desktop/tablet/mobile) และผลต่อ spacing/type/line-height
- เดโมเร็ว ๆ กับปุ่ม/เฟรมเพื่อให้เห็นการเปลี่ยนค่าตามโหมด
- สรุปทิศทางว่าค่าที่วางไว้วันนี้คือฐานสำหรับต่อยอดไปตอนสร้างคอมโพเนนต์จริง

## Key Takeaways (10 ข้อ)
1. เริ่ม Design System ที่ "token architecture" ก่อนคอมโพเนนต์เสมอ
2. แยกชั้น `brand -> alias -> mapped` เพื่อควบคุม abstraction และการเปลี่ยนแปลง
3. ใช้ semantic tokens (เช่น info/success/warning/error) เพื่อให้ UI อ่านความหมายได้
4. ตั้งชื่อ tokens/styles ให้คงเส้นคงวาเพื่อลดต้นทุนการสื่อสารในทีม
5. เลือกกริดที่ใช้งานจริง (เช่น 4px) แล้วให้ type/spacing เดินตามระบบเดียวกัน
6. Modular scale เป็นจุดเริ่ม ไม่ใช่กฎตายตัว ต้องปรับให้เข้ากับผลิตภัณฑ์จริง
7. ค่า line-height และ paragraph spacing ควรตัดสินจาก readability + accessibility
8. บางหมวด (เช่น border width/radius) อาจไม่ต้องมี mapped layer ถ้าไม่เพิ่มคุณค่า
9. รองรับ responsive ตั้งแต่ระดับ token จะช่วยลดงานแก้ตอนขยายระบบ
10. ทำเวอร์ชันใช้งานได้ก่อน แล้ว iterate จากการใช้งานจริง คือแนวทางที่เร็วและเสถียร
