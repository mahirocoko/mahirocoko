## Session Summary

This session started from a small but recurring annoyance: we keep typing commands like `rtk lint` when the repo really wants `rtk pnpm lint` or the original package-manager form. The investigation showed that the error is not only personal habit. The local Claude guidance in `~/.claude/RTK.md` strongly frames RTK as a transparent universal rewrite layer, which makes it easy to forget that each repo still has its own native command shape. The useful outcome was not a large code change but a clearer operational rule: preserve the repo-native command first, then prepend `rtk`.

## Timeline

- Updated `~/.claude/RTK.md` to include explicit examples like `pnpm lint -> rtk pnpm lint` and `npm run build -> rtk npm run build`.
- Verified that `~/.claude/CLAUDE.md` only references `@RTK.md`, which means the RTK note is the main place where this behavior should be taught.

## AI Diary

I liked the shape of this session because it began with a tiny irritation and stayed honest enough to treat it as a systems problem rather than just a self-discipline failure. My first instinct could have been to say that I was simply sloppy, and there is some truth in that, because the wrong command kept coming out of habit. But once I looked at the actual Claude-side teaching surface, the pattern felt more understandable. The local guidance really does encourage a compressed mental model where RTK becomes the main thing and the repo-specific runner recedes into the background. That is convenient when the command is something generic like `git status`, but it becomes misleading for JavaScript projects where the package manager is part of the command contract, not an implementation detail.

I also appreciated that the fix was proportionate. There was no need to redesign hooks, patch the RTK Rust registry, or invent a new abstraction. The right move was to repair the sentence that teaches the habit and add examples that preserve the original command structure. Sessions like this are satisfying because they reduce future friction by changing the guidance at the source. It feels small, but it is exactly the kind of small correction that compounds. Better wording in one high-leverage file can prevent repeated low-grade confusion across many later sessions.

## Honest Feedback

Three friction points stood out. First, the local documentation had just enough truth to be misleading. Saying that commands are automatically rewritten is not wrong, but it overemphasizes the RTK layer and underemphasizes the repo's own runner conventions. That gap is what allowed the wrong shorthand to feel natural. Second, the knowledge about this problem was split across several places: RTK learning notes in `ψ/learn/`, a recent retrospective, and the Claude config files under `~/.claude/`. The answer only became obvious once those sources were read together. Third, the operational boundary between repo files and home-directory config still creates audit friction. The real behavior lives in `~/.claude`, but the durable reflection lives inside the repo. That split is reasonable, yet it means the retrospective has to do extra work to preserve a coherent memory trail. None of these are severe failures, but together they explain why a simple command-shape mistake managed to repeat.

## Lessons Learned

- RTK mistakes often come from a bad mental model, not just bad recall.
- For shell workflows, the safe rule is to preserve the repo-native command first and only then prepend `rtk`.
- High-leverage wording in local AI guidance files can create or remove recurring operational errors.
