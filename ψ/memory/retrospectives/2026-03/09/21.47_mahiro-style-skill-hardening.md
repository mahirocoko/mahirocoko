## AI Diary

This session felt like a proper external audit. Mahiro asked me to study how the best in the industry write agent skills — Vercel, Anthropic — and then turn that lens back on the mahiro-style skill that lives in this repo. What I found surprised me: mahiro-style is genuinely better-designed than the reference repos in several ways. The ownership declarations ("this page owns X"), the explicit anti-ownership ("do not use this page for Y"), the 4-level precedence system — none of the reference repos have these. Those are real innovations in skill design.

But the benchmarking also revealed honest gaps. Three pattern pages had zero code examples, relying entirely on prose descriptions while every reference repo makes Bad/Good code pairs a non-negotiable. The lack of machine-friendly detection triggers was another blind spot — next-skills uses explicit "Detect:" patterns that help agents auto-activate, while mahiro-style only had soft "Use it when..." prose.

The validation step was the most important. Instead of trusting that the doctrine was right, Mahiro asked me to actually read the 3 codebases that the doctrine was written from. That discipline — checking whether the map matches the territory — is rare. The results confirmed the changes: every Detect trigger I wrote corresponds to real code patterns in at least one of the repos. The code examples in components.md, hooks.md, and best-practices.md are grounded in actual patterns from eizypay-fe and jit-flow, not hypothetical scenarios.

Two minor gaps remain (thick CRUD list pages as valid convention, queryOptions factory pattern), but these are additions, not corrections. The doctrine doesn't contradict reality — it just hasn't acknowledged every valid variant yet.

## Honest Feedback

**Friction 1: Token budget on legacy redirects.** Reading all 36 files in the mahiro-style skill was expensive. 17 of those are legacy redirect shims in `resources/` and `examples/` that contain almost no content. They exist for "Nothing is Deleted" compliance, but they burn context tokens every time an agent navigates into them. A `.agentignore` or explicit "do not load" directive in SKILL.md would help.

**Friction 2: Parallel analysis hit tool limits.** Analyzing 3 reference repos + mahiro-style simultaneously required 4 parallel Task agents, each reading dozens of files. The results were good but some agents hit the point where they had to summarize rather than quote exact content. A pre-indexed skill metadata format (like vercel's `metadata.json`) would make future comparisons faster.

**Friction 3: Validation required re-reading after changes.** After implementing the 3 improvements, validating against real codebases meant re-analyzing files I'd already studied. The session would have been tighter if the validation step came first — analyze codebases, then implement changes. Instead the flow was: study references → implement → validate. The validate step confirmed everything was fine, but the ordering created redundant work.

## Lessons Learned

- mahiro-style's layered architecture (foundations → patterns), ownership declarations, and precedence system are innovations not found in Vercel or Anthropic skill repos
- Every pattern page should have both code examples and machine-readable Detect triggers — prose-only guidance is weaker for agent consumption
- Doctrine should be validated against actual codebases, not just designed from memory of them
- Decision trees provide faster ownership routing than prose descriptions for cross-cutting questions
