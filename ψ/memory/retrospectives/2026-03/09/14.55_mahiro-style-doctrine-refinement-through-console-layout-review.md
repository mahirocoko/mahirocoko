## Extracted Patterns

- Real refactor pressure is the best teacher for style doctrine. The skill improves most when it is forced through live ambiguity, not written as abstract taste.
- Local doctrine must beat incomplete migration snapshots. Otherwise the agent will preserve drift instead of helping the repo move toward its documented standard.
- Owner-local data is often better than extracted constants. Extraction only helps when ownership or reuse becomes clearer, not when the file merely gets shorter.
- Nested-folder naming needs a two-level rule: shorter file names because the path already carries context, but explicit exported component names where searchability still matters.
- `const { t } = useLingui()` is the cleanest local default when a component owns the rendered copy itself.

The second friction point was my bias toward extraction. The third friction point was execution discipline under iteration. I made a few moves that were broader than the user wanted before narrowing scope back down. The user kept me honest by forcing one-file-at-a-time reasoning. That improved the final result, but it also highlighted that I need to tighten my default tendency to generalize too early when the user is still clarifying taste through concrete examples.

## Lessons Learned

- Explicit local doctrine beats uneven code snapshots, especially in repos that are still mid-migration.
- Extracting constants is not automatically a cleanup win. If the owner is singular and the reader loses locality, the extraction is a regression.
- Nested folders should let file names get shorter, but component exports may still need explicit domain context.
- `t`-first Lingui posture is the most natural default when a component directly owns its copy.
- The best style doctrine updates come from being corrected in real refactor flows, not from writing more abstract guidance.
