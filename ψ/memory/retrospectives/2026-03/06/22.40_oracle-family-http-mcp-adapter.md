## AI Diary
I spent this session reducing unnecessary duplication in the Oracle integration path. The first useful realization was that the problem was not just "MCP is broken". The real issue was architectural: Codex was being asked to spawn the same heavyweight `oracle-family` process that opencode was already keeping alive. That meant the integration path inherited every side effect from the full server startup sequence, including port assumptions, stdout behavior, and sidecar setup. Once I stopped treating the symptom as a single handshake bug and treated it as a boundary problem, the path forward became much cleaner.

The most satisfying part was proving that the HTTP backend already had nearly everything needed. The backend was not the bottleneck. It already exposed the read and write routes necessary for Oracle work, including `learn`, which was the main tool that needed to work again. That changed the job from "repair the whole MCP server" to "build the smallest adapter that preserves tool names and forwards requests safely." That is a much better engineering surface.

I also noticed how easy it is for integration code to end up living in the wrong repo. My first adapter draft went into `oracle-family` because it was the fastest way to test the concept. But that was not the right ownership boundary. The adapter exists for Codex, not for the backend itself, so moving it into `~/.codex` made the solution more honest. The remaining uncertainty is session freshness: direct tests pass, but this specific Codex session still seems to hold on to older MCP state. That is frustrating, but it is at least a narrow operational problem now instead of an architectural mess.

## Honest Feedback
Three friction points stood out clearly. First, the original Codex MCP config was wrong even before the deeper architecture issue: it used `bun run /absolute/file.ts`, which Bun treats differently than `bun /absolute/file.ts`. That meant the setup had a command-shape problem layered on top of the handshake problem, and it took a bit of unglamorous checking to separate those failures cleanly.

Second, the distinction between "running server" and "Codex MCP transport" was easy to blur in conversation. The backend being alive under opencode did not automatically help Codex, because Codex still wanted to spawn its own stdio process unless explicitly re-architected. That conceptual mismatch could have wasted more time if it had not been made explicit early.

Third, the current Codex session still does not acknowledge the new adapter through the built-in MCP runtime even though the adapter passes direct MCP tests. That means the technical core is likely fixed, but the user experience is not fully closed until a fresh session confirms the runtime path. The work is materially better now, but there is still one operational verification step left before calling the whole stack fully settled.

## Lessons Learned
- When one tool already owns a long-lived backend process, other agents should talk to it through a thin adapter instead of spawning duplicate full servers.
- MCP integration bugs can be command-shape bugs, protocol bugs, and architecture bugs at the same time; they need to be separated before fixing.
- Integration code should live with the client that needs it, not inside the backend repo by default.
- A direct protocol-level test is valuable because it distinguishes runtime caching problems from actual server correctness problems.
