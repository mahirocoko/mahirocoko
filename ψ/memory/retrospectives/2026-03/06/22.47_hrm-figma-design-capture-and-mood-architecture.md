## AI Diary
I spent this session turning an awkward tooling mismatch into something useful. At the start, the Figma ask looked deceptively simple: “design in Figma.” But the actual tool boundary mattered a lot. The MCP connection could read nodes, implement from nodes, and import captured pages, yet it could not behave like a full design assistant that directly draws and rearranges layers inside the file on command. Once I stopped fighting that boundary and treated it as a constraint to design around, the work became much cleaner.

The most satisfying part was that the product reasoning and the tooling workaround reinforced each other instead of competing. Building the screens as local HTML/CSS artifacts let me iterate on the structure honestly. I could see that the first mood tracker placement was wrong for a dashboard. It felt like a form interrupting a summary surface. That pushed me toward the better product split: clock-in owns mood entry, dashboard owns mood reflection. Because the source was local and editable, I could change the information architecture, refine the visual density, and then push the updated result straight back into Figma without pretending the original node-based workflow was sufficient.

I also liked that this session produced multiple kinds of durable memory at once: the Figma file itself, the local prototype source, the design rationale, and now this retrospective. That fits the repo’s bias toward preserved trails instead of neat but forgetful cleanup. The one thing that still feels unfinished is ownership of the next stage. The mockups are coherent now, but they still need either a stronger token/state spec or a deliberate move into implementation.

## Honest Feedback
Three friction points stood out. First, the Figma MCP affordances were easy to misread at the start. “Connected MCP” sounds like full design control, but in practice the most reliable path here was reading, importing, and capturing, not directly editing the design file as if it were a programmable canvas. That mismatch cost time until I made the boundary explicit.

Second, the shared node links were not initially actionable. The first URLs pointed at a page root or empty frame, which meant the design context responses were technically valid but practically useless. That kind of false start is annoying because it creates motion without insight. It reinforced that for node-based work, the exact target matters a lot more than the existence of a Figma file.

Third, the product decision around mood tracking was initially too literal. The first instinct was to add the input straight onto the dashboard because that was the screen already in motion. But that would have weakened the dashboard’s job and made the product feel noisier. The correction was the right one, but it showed how easily UI composition can drift away from task logic when the design surface is moving quickly. Next time I should challenge “where does this ritual belong?” earlier, before drawing it at all.

## Lessons Learned
- When direct Figma canvas authoring is limited, local HTML/CSS plus MCP capture is a practical bridge that keeps design iteration fast and editable.
- Mood input belongs in the ritual that creates it, not on the dashboard that summarizes it.
- A dashboard should show today’s state and monthly trend, while the action flow should collect the underlying data.
- Narrow `.gitignore` rules are better than ignoring all of `lab/` because they preserve future flexibility without polluting current status.
