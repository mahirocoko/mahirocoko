## AI Diary
I spent this session trying to correct a subtle but important mismatch: the `frontend` skill already carried a lot of Mahiro's taste, but it still presented itself like a command-driven frontend helper instead of a doctrine loader. Once that clicked, the whole session became less about "improving docs" and more about restoring alignment between identity and interface. I could see that the earlier design had useful pieces, but it was asking the user to think like a tool author, not like Mahiro in conversation. That was the wrong center of gravity.

What helped most was grounding the abstract discussion in actual repos. `haabiz-hrm-fe` showed the clean, distilled version of the style: token-first, route orchestration, provider boundaries, reusable UI contracts. `jit-flow` showed the heavier, more integrated version: theme systems, AntD hybrid decisions, service/store/hook layering, and deeper operational docs. Looking at both made it obvious that "Mahiro style" is not one exact component library choice. It is a consistent way of controlling complexity. That insight made the refactor feel justified rather than cosmetic.

I also noticed how much wording matters for agent behavior. A README that starts with flags and categories quietly teaches the wrong usage pattern. Once I rewrote the docs around natural prompts such as "ช่วย refactor ให้เป็นในแบบของฉัน", the skill started feeling much more honest. I left the retrieval system intentionally simple, but the shape is now closer to the real thing: the skill loads judgment first, then lets the AI apply it. That feels like a better foundation to keep evolving from.

## Honest Feedback
Three friction points stood out. First, the skill had drifted into an awkward middle state where it wanted to feel intelligent and conversational, but its retrieval model was still mostly text matching. That created false expectations in the docs and examples. The refactor improved the framing, but the underlying retrieval is still noisy in places, especially around code fences and example syntax lines. The next iteration should focus on cleaning that signal up rather than adding more surface area.

Second, the doctrine is currently spread across portable resources, repo-local `AGENTS.md`, and lived examples in real projects. That is powerful, but it also means the truth can fragment. I saw this clearly in `jit-flow`, where some documentation still reflects older persistence assumptions while the live store code has already moved to Supabase session synchronization. The skill can only be as current as the doctrine sources it reads. If the source of truth is not pruned, the skill will slowly become a historian instead of an operator.

Third, the README got much better, but it is still carrying a lot of explanation because the skill is doing two jobs at once: teaching the concept and giving usage examples. That is acceptable for now, but there is a risk of making the docs feel heavier than the intended interaction. A future pass may need a sharper split between "understand the doctrine" and "copy these prompts now." The important thing is that the core direction is right now; the remaining friction is mostly about sharpening, not rethinking.

## Lessons Learned
- A personal AI skill should be framed around identity first, commands second.
- Natural-language prompting should be the default interface when the skill's main job is retrieval and steering.
- Real project patterns (`haabiz-hrm-fe`, `jit-flow`) are more valuable than abstract best-practice prose when defining doctrine.
- Optional lenses work best as precision controls, not as the required entry point.
