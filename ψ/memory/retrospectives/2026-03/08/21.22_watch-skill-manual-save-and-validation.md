## AI Diary

I felt this session expose a familiar tension between workflow theater and workflow truth. On the surface, `/watch` looks like a simple automation: send a YouTube URL to Gemini, get a transcript, save a note, move on. But the moment I actually ran it, the difference between a command being dispatched and a result being durable became the whole story. I had to slow down and resist the temptation to call the job done just because a browser tab opened and a script printed success. The useful part of the session came from checking the actual Gemini tab state, confirming the conversation URL, reading the returned text, and making sure there was something worth preserving.

What made the session feel worthwhile is that the friction was not random. It revealed a structural truth about this kind of toolchain: the most fragile parts are the handoff boundaries. The repo path assumed by the skill was wrong for the current working tree. The save helper had a latent bug. The Gemini step could easily have been treated as successful before any transcript body was verified. Each of those is small on its own, but together they explain why memory systems either become trustworthy infrastructure or noisy archives. I am glad the session ended with a clean retained note and an Oracle learning instead of a false positive.

## Honest Feedback

First friction point: the skill documentation and the local runtime shape were slightly out of sync. The instructions assumed a script path that did not exist in the current repo, which meant the first execution failed before the real work even began. That is the kind of mismatch that makes a workflow feel more polished than it actually is. Second friction point: the save path inside the skill is not production-safe yet. `save-learning.ts` appears to reference `SLUGS_FILE` without defining it, so the official retention step cannot be trusted blindly. A knowledge-capture workflow that breaks at the final persistence step is especially dangerous because it gives the illusion of completion. Third friction point: response validation is still too manual. I had to query the Gemini tab directly to confirm the generated output and conversation URL. That works, but it means the skill still depends on an operator noticing whether the model actually answered in a usable form.

More broadly, this session reinforced that durable automation needs explicit validation at every seam: path resolution, model completion, fallback capture, and final save. Without that, the system is smooth only in demos.

## Lessons Learned

- A `/watch` run is only complete when the Gemini response body and conversation URL are both captured, not when the prompt dispatch succeeds.
- Fallback captions are not just a backup artifact; they are a verification layer that keeps the learning pipeline useful even when Gemini output is partial.
- Retention helpers must be treated like production code, because one broken variable can silently collapse the last mile of a memory workflow.
