## Honest Feedback

The first friction point was orchestration overhead. I used specialist flows aggressively, which was correct in principle, but it also increased latency and let one visual skill bias the first implementation too far away from the reference. The lesson is not “do less delegation”; it is “delegate with tighter aesthetic constraints when fidelity matters.” If the requirement is reference-first, then a general frontend beautification skill is dangerous even when the resulting UI looks good.

The second friction point was system inconsistency around configuration and docs. `oh-my-opencode` currently has documentation that still shows an `lsp` block while the active schema rejects it. That kind of drift wastes focus because it turns a clean config update into source archaeology. I was able to resolve it, but the path there was noisier than it should have been.

The third friction point was content model drift in the landing page work. Even after the faithful second pass, the page still wanted to explain the redesign instead of behaving like a real landing page. That is a subtle but recurring problem: AI-generated UI often talks about itself. Catching that required a second review pass that focused on product framing rather than colors or spacing. I need to keep that trap in mind more explicitly when a page is supposed to sell, guide, or structure rather than narrate its own construction.

## Lessons Learned

- For reference-led UI work, disable or avoid skills that optimize for originality over fidelity.
- Preserve page decomposition and code shape, but be willing to rollback the entire visual thesis if it diverges from the reference language.
- A faithful landing page is mostly about section order, value-proposition copy, and repeated card/CTA motifs; color cleanup alone is not enough.
