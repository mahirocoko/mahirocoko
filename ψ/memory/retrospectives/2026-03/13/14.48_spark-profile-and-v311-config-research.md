## Honest Feedback
The biggest friction point was tool behavior around Oracle consult and background tasks. Oracle search behaved inconsistently for this request, so I had to switch from a direct consult path to lighter Oracle-backed reflection and then continue with grounded evidence from the repo and web. That was manageable, but it broke flow. The second friction point was that model support in this ecosystem is fragmented across official docs, cached provider lists, issue threads, and local one-off overrides. It takes more time than it should to answer a basic question like “is this model safe to use?” because the answer depends on naming, access tier, and runtime behavior, not just existence. The third friction point was profile discoverability. The config filename remains `oh-my-opencode.json` even after the project rename, which is understandable for compatibility, but it makes variant management clumsy unless you already know about `OPENCODE_CONFIG_DIR`. The final result was solid, but the surrounding ergonomics still ask the operator to carry too much implicit knowledge.

## Lessons Learned
- A real model identifier is not automatically a safe global default; availability and runtime support still need to be verified.
- For experimental models, the right pattern is a contained profile with explicit fallbacks, not a mutation of the primary working config.
- Release alignment work often becomes simpler when you compare current local state first instead of assuming a migration is necessary.
