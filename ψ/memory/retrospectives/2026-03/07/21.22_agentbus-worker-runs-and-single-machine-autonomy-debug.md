# Session Retrospective: agentbus-worker-runs-and-single-machine-autonomy-debug

**Date**: 2026-03-07 21:22 +07  
**Type**: Implementation + debugging retrospective  
**Primary Project**: `agentbus`  
**Branch**: `main`  
**Session Focus**: Push AgentBus closer to seamless single-machine autonomous collaboration between `codex` and `opencode`, then make the runtime observable enough to trust and debug.

## Session Summary

This session moved AgentBus from "interesting orchestration prototype" toward "something a human can actually operate and inspect." The initial pressure came from one repeated pain: even when `codex` and `opencode` seemed to coordinate, it was still too hard to tell whether anything autonomous had really happened or whether the system was just moving messages around and relying on human prompts.

The work therefore split into three layers:

1. **Autonomous thread semantics**  
   The system had already grown durable tasks and worker pickup, but it still behaved like a task bus more than a conversation bus. We pushed the model toward `autonomous_loop` threads by adding thread ownership and wake rules. That let a worker be treated as the owner of an ongoing conversation rather than only the owner of a single claimed task.

2. **User-facing command surface**  
   The system still felt too plumbing-heavy for end users. A front-door CLI was added so the happy path could be described as `start`, `status`, `doctor`, and `smoke` rather than a pile of low-level commands and hand-shaped payloads.

3. **Runtime observability**  
   The most important practical shift was adding persistent worker-run telemetry. Before this, it was possible to theorize that `codex` or `opencode` had been spawned. After this, the system could record launch intent, command shape, trigger, exit status, and run summary. That changed debugging from guesswork into inspection.

The biggest conceptual realization of the session was that autonomy is not just a transport problem. Even once the system can wake workers by thread rather than by task, humans still do not trust it unless they can see concrete evidence that the runtime actually launched and what happened when it did.

## What Changed

### 1. Thread ownership became a first-class concept

The core model was extended so a thread can carry:

- `mode`
- `ownerPeerId`
- `wakeupPolicy`
- `stopCondition`

This was the necessary step to move beyond "task gets completed, then the conversation dies" toward "a worker owns the thread and can be woken again when follow-up arrives." The corresponding wake logic was pushed into the agent daemon so that `note`, `result`, `question`, and `fail` messages can resume the owner even when `taskId` is `null`.

This directly addresses the previously identified gap: the system used to wait on tasks, not on threads.

### 2. AgentBus gained a simpler operator surface

The repo received a front-door CLI with commands shaped for actual use:

- `agentbus:start`
- `agentbus:status`
- `agentbus:doctor`
- `agentbus:smoke`
- `agentbus -- runs`

This matters because the project keeps trying to become "AI-native," but humans still need a clean way to inspect and bootstrap the system. Good autonomy still requires good operator ergonomics.

### 3. Worker launch telemetry became durable

This was the most practically important change of the session. A new worker run record was added so the system can persist:

- which worker was launched
- which runtime it used
- why it was launched
- which command was prepared
- whether it parked, completed, or failed
- its exit code and summary

The CLI was then extended so recent launches can be queried by thread or globally. This means the question "Did `codex` or `opencode` actually spawn?" finally has an evidence-based answer.

## Verification Performed

The work was verified in isolated local test setups rather than only by reasoning:

- `bun run check` passed after the major changes
- temporary servers and temporary databases were used on alternate ports
- `agentd --dry-run` was used to validate lifecycle changes without burning real runtime sessions
- owner-thread wake behavior was demonstrated:
  - root request sent to `mahiro.codex.worker`
  - task claimed and parked
  - follow-up `note` with `taskId = null` sent into the same thread
  - the thread owner was resumed by the daemon
- command preparation for both runtimes was observed:
  - `codex exec --skip-git-repo-check -C ...`
  - `opencode run --dir ...`
- worker run telemetry was exercised through CLI commands to show durable launch records

This was enough to prove the orchestration mechanics and the observability path, even though the deepest live inference path was intentionally kept in dry-run mode during validation.

## What the Real Thread Revealed

The live inspection of `thr_78fa38e4038a49e1b4fb7d6550cdcb58` produced an important reality check.

The thread showed that:

- `mahiro.opencode` opened a root task to `mahiro.codex.worker`
- `mahiro.codex.worker` delegated a subtask to `mahiro.opencode.worker`
- both tasks completed
- the final summary returned successfully

So the system already supports real conductor-specialist orchestration.

But it also showed:

- this was still not a free multi-turn dialogue
- some `note` messages remained queued
- the live thread response did not include the new ownership metadata

That last point matters a lot. It strongly suggests the process running on `127.0.0.1:47780` was still serving an older build. In other words, one of the hardest debugging problems was not in the code itself, but in the mismatch between updated source and stale running processes.

This is a classic distributed-systems lesson in miniature: the truth is not what the repo says; the truth is what the live process is actually running.

## Emotional / Cognitive Tone

The session had a very familiar engineering rhythm:

- excitement about the architecture
- friction when the system still felt less autonomous than intended
- a sharper reframing of the real problem
- relief once observability improved enough to stop arguing with shadows

The deepest psychological shift was from "maybe it spawned" to "we can prove whether it spawned." That single move reduced a lot of ambiguity in the project.

There was also a subtle correction to the product perspective. The user repeatedly pushed toward end-user simplicity, not just runtime cleverness. That pressure was healthy. It forced the work away from internal elegance only, and toward command surfaces and debug surfaces that a real human can actually trust.

## Honest Feedback

### What went well

- The session identified the true gap cleanly: not just task ownership, but thread continuity.
- The implementation work stayed close to observed pain instead of wandering into abstract architecture.
- The new telemetry materially improved trust in the system.
- The CLI surface makes it easier to imagine how an AI agent would front this for a user.

### What was difficult

- The live process mismatch on the main port made the system feel more inconsistent than the source code actually was.
- Without durable run telemetry, the debugging story was too speculative.
- The system still needs one more abstraction pass before it feels naturally "auto" to a non-technical user.

### What still feels unresolved

- The current success case is conductor-specialist orchestration, not true open-ended multi-turn agent conversation.
- The UI now helps with visibility, but runtime restart state still needs to be much more obvious.
- The end-user flow still depends on humans understanding too much about workers and routing.

## Pattern Observed

When building collaborative agent systems, the first implementation instinct is to over-focus on transport and protocol shape. But the thing users actually notice is much simpler:

- Does the right worker wake up?
- Can I see that it woke up?
- Can I see what it tried to do?
- Can I tell whether the live system is running the code I think it is?

Autonomy is experienced through observability.

## Closing Reflection

The session did not fully solve seamless autonomy, but it significantly reduced ambiguity around why autonomy feels weak and how to debug it. That is real progress. The system now has a better model for ongoing thread ownership and a much stronger proof surface for runtime activity.

That combination matters. A collaborative agent system does not become trustworthy just by being clever. It becomes trustworthy when its behavior can be seen, explained, and reproduced.
