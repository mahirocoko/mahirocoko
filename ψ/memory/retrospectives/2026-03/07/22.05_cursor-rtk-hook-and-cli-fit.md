## Session Summary

This session started as a practical question rather than an architecture exercise: is `rtk-ai/rtk` actually worth adopting for a workflow centered on `opencode`, `oh my opencode`, and Cursor CLI? The first half of the work was research-heavy. A deep learning pass was created under `ψ/learn/rtk-ai/rtk/`, covering architecture, code snippets, quick reference, testing, and API surface. That shifted the repo from a vague “token saver” impression into a concrete internal model: RTK is a Rust CLI proxy with a thin Clap router, many command-specific adapters, a tracking database, and a hook-first integration philosophy.

The second half of the session focused on Cursor compatibility. Claude already had a working `PreToolUse` RTK hook, so the main question was whether Cursor offered a comparable event. Once Cursor hooks documentation and examples were checked, it became clear that the closest fit was `beforeShellExecution`, not `preToolUse` in the Claude sense. That led to a small but important product translation: Cursor does not appear to expose the same `updatedInput` rewrite protocol, so the usable fallback is a deny-and-retry pattern. The resulting hook does not silently rewrite the shell command. Instead, it blocks the original command and instructs the agent to retry with the exact `rtk ...` replacement.

The session therefore produced both understanding and artifact: a documented learning set for RTK and a functioning Cursor-side RTK hook configuration.

## AI Diary

I felt this session settle into a satisfying shape because it stayed anchored to a real operational question instead of drifting into general tool enthusiasm. At the beginning, RTK could easily have become one more interesting utility that sounds clever but never actually earns a place in the daily workflow. The deep learn pass helped strip away that fuzziness. Once I saw the internal structure, especially the hook-first model and the tracking layer, the value proposition became much easier to judge honestly. This was not just a pretty README. It is a serious attempt to shape shell output for LLM-first work.

The most useful emotional turn in the session was the move from “can Cursor do something like Claude?” to “what is Cursor’s actual hook contract, and what is the minimal translation that respects it?” That shift matters. It stopped me from trying to force the Claude model onto Cursor unchanged. I had to accept that Cursor likely does not offer the same transparent input rewrite surface here, and that the proper engineering response is not denial but adaptation. The deny-and-retry flow is less elegant than silent rewrite, but it is concrete, debuggable, and aligned with the available mechanism.

I also appreciated the small recovery loop when the hook was edited midstream. That could have turned into confusion or needless blame. Instead, it became a tiny proof that the work was grounded enough to rebuild cleanly from observed state. Sessions like this are less glamorous than designing a whole new system, but they are exactly how operational trust gets built: inspect the real integration surface, port only what the platform can truly support, verify the behavior, and document the tradeoff honestly.

## Honest Feedback

Three friction points stood out clearly. First, the public docs around Cursor hooks were enough to establish direction but not enough to prove the exact rewrite envelope. That forced a more conservative implementation than I would prefer. I could build something plausible and testable, but not something guaranteed by a crisp schema contract. Second, the existing Claude hook created a temptation to assume near-drop-in compatibility. It was useful as a reference, but it also increased the risk of carrying over protocol assumptions that do not belong in Cursor. The temporary user edit that pointed Cursor directly at the Claude script made that mismatch more visible. Third, part of the work happened outside the repo in `~/.cursor`, while the learning artifacts lived inside `ψ/learn/`. That split is operationally correct, but it weakens the sense of a single audit trail unless the retrospective ties the two halves together explicitly.

There is also a broader process critique here. I relied on a combination of repo inspection, docs snippets, and live payload simulation rather than one authoritative Cursor schema source. That is acceptable for a pragmatic first integration, but it is still a compromise. If this hook becomes part of daily workflow, it deserves one more pass with stronger source confirmation and a stricter allowlist. The current solution is useful, but it is still an informed adaptation rather than a formally proven integration.

## Lessons Learned

- RTK is worth considering for AI-heavy terminal workflows because it combines broad command coverage with a hook-first operational model, not just ad hoc wrappers.
- A Claude hook script is reusable as logic, not as protocol. The `rtk rewrite` core can be carried across tools, but the event envelope must match the host platform.
- For Cursor shell hooks, the practical fallback is deny-and-retry when native transparent command mutation is not available.
- Research artifacts in `ψ/learn/` materially improve decision quality before doing local tool integration work.
