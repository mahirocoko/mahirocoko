## Honest Feedback
มี friction อยู่ชัดเจนอย่างน้อยสามจุดใน session นี้ จุดแรกคือการวนกับ Figma capture หลายรอบเกินความจำเป็นเพราะเราเพิ่งมารู้ภายหลังว่า local URL path ไม่ตรงกับ working directory ของ server นี่เป็นความผิดพลาดแบบพื้นฐาน แต่กระทบเวลาและทำให้เราตีความอาการ pending ผิดช่วงหนึ่งว่าเป็นปัญหาเรื่อง icon หรือ converter ล้วนๆ ทั้งที่จริงมีเรื่อง path ปนอยู่ด้วย

จุดที่สองคือเรื่อง icon system ฉันปล่อยให้มีหลายวิธีอยู่พร้อมกันนานเกินไป ทั้ง image, CSS mask, และ pseudo-inline approach การตัดสินใจให้เด็ดขาดเร็วกว่านี้จะลดงานแก้ซ้ำและทำให้หน้าอื่นไม่ต้องตามมาแปลงย้อนหลังทีหลัง Session นี้เลยเตือนชัดว่า design prototype ที่ต้อง sync กับ external tool ควรเลือก representation ที่ robust ตั้งแต่แรก มากกว่าที่จะเลือกวิธีที่สะดวกเฉพาะใน browser

จุดที่สามคือแม้เราจะสร้างหลายหน้าและหลายเวอร์ชันได้เร็ว แต่ความเร็วนี้ก็ทำให้เกิดการกระจายตัวของ artifact ค่อนข้างมาก ถ้าไม่รีบคัด candidate หลัก มันจะเริ่มรกทั้งใน local และใน Figma เอง ฉันคิดว่ารอบหน้าเราควรตัดสินใจเรื่อง pruning, naming, และ promotion path ให้ชัดกว่านี้ เช่น หน้าไหนคือ exploration, หน้าไหนคือ candidate, หน้าไหนคือ source of truth เพื่อไม่ให้แต่ละรอบของการ iterate เพิ่มภาระในการจัดการงานย้อนหลัง

## Lessons Learned
- ถ้าต้องการให้ Figma HTML capture เห็น icon ชัดและแปลงได้สม่ำเสมอ ควรใช้ inline SVG พร้อม path ตรงๆ ใน DOM
- เวลาจับปัญหา capture ที่ค้าง `pending` ต้องเช็ก server root และ local URL path ก่อนสรุปว่าเป็นปัญหา converter
- Homepage ของ HRM ที่เน้น wellbeing ควรเล่า “rhythm ของวันและช่วงพัก” มากกว่าจะยัด operational metrics ให้ครบทุกอย่าง
- การแยก `tokens.css` กับหน้า `design-system.html` ช่วยให้ prototype ขยับจาก exploratory mockup ไปเป็น system-aligned mockup ได้เร็วมาก
- การมีหลาย variant มีประโยชน์ตอน explore แต่ต้องรีบ converge เป็น candidate เดียวก่อน artifact จะกระจายเกินควบคุม
