## Honest Feedback

1. **Bun cache fragility**: This is the third time Bun's global cache has caused ENOENT errors on transitive dependencies. The fix is always "nuke everything." There should be a `bun cache clean` that actually works, or the watch/gemini scripts should pin their dependency resolution more carefully. Time wasted: ~3 minutes.

2. **Gemini response timing is a guess**: I waited 45 seconds then checked, but the response was still "Connecting...". Then waited 30 more seconds and it was done. There's no callback or polling mechanism — just sleep-and-pray. A `get-response --wait` flag that polls until content appears would eliminate this guesswork.

3. **The `/watch` scripts live in global skills but need workspace `node_modules`**: Running `bun /Users/mahiro/.cursor/skills/watch/scripts/transcribe.ts` worked because it uses the workspace's mqtt dependency. But `get-response.ts` from the gemini skill hit the same dependency issue. The skill scripts should either bundle their dependencies or have a clear install step.

## Lessons Learned

- Skills-as-context is the real differentiator for AI frontend design, not model choice alone (confirmed by both Theo's findings and our existing doctrine)
- Bun global cache corruption: when one package fails to resolve, nuke `~/.bun/install/cache` entirely rather than trying to fix individual entries
- Iteration ability (model listens to feedback and refines) matters more than first-shot quality for design workflows
