## Key Code Changes

### Source Language Pattern (the key insight)

```tsx
// ❌ WRONG - English source when DEFAULT_LANG = 'th'
const { t } = useLingui()
return <h1>{t`Sign in`}</h1>

// ✅ CORRECT - Thai source when DEFAULT_LANG = 'th'
const { t } = useLingui()
return <h1>{t`เข้าสู่ระบบ`}</h1>
```

### Constants Don't Need Translation

```tsx
// ✅ Use APP_NAME directly
import { APP_NAME } from '@/constants/config'
return <p>{APP_NAME}</p>

// ❌ Don't wrap constants in t
return <p>{t`Haabiz HRM`}</p>
```

## AI Diary

This session taught me something fundamental about i18n that I had backwards. I initially wrote all the `t` macro strings in English, thinking "extract to English, translate to Thai." But when the default language is Thai, the *source* should be Thai. The `t` macro wraps what you actually write in your code, not what you want to translate *from*.

This clicked when the user asked why `Haabiz HRM` was using `t` instead of `APP_NAME`. Two realizations: (1) constants don't need translation, and (2) if the default language is Thai, the source strings in code should be Thai. The `.po` files then contain the source (Thai) and translations (English).

I also appreciated the user's patience when I made the JSX structure error — duplicate `</CardContent>` tags from the edit operation. The lingui CLI caught it during extraction, which was actually helpful validation.

The pattern extraction into the skill felt natural. "If I learned this, future agents should too." The i18n.md went from a basic example to a comprehensive guide with folder structure, gitignore patterns, provider integration, and the core source language rule.

## Honest Feedback

1. **Edit tool fragility**: The batch edit created duplicate `</CardContent>` tags. I should have either done smaller edits or verified the result immediately. The lingui CLI caught it, but it could have been a runtime bug.

2. **Assumption about source language**: I assumed "English is source, Thai is translation" without checking `DEFAULT_LANG`. The config clearly said `'th'`. Should have read constants first before writing `t` strings.

3. **Constants vs translation confusion**: Initially wrapped `Haabiz HRM` in `t` macro. Should have recognized it as a brand/constant that doesn't change across locales.

## Lessons Learned

1. **Source language = DEFAULT_LANG**: The `t` macro wraps the default language. If `DEFAULT_LANG = 'th'`, write Thai in `t`. The `.po` files contain source (default) + translations (other languages).

2. **Constants skip translation**: `APP_NAME`, brand names, proper nouns — use directly, don't wrap in `t`.

3. **Compiled locale files are artifacts**: `app/locales/*.js` are generated by `lingui compile`. Only track `.po` files, ignore `.js`.

4. **jit-flow is the reference**: For Haabiz projects, jit-flow has production patterns to copy. Use it as template.
