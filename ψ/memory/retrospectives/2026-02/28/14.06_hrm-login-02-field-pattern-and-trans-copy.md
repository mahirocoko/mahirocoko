The durable approach is to lock one canonical structural source and consciously treat the other as visual reference only.

By the end, I felt the session regained discipline. The resulting code is cleaner, more predictable for future contributors, and closer to what we have repeatedly learned: boundary-layer fidelity is not optional.

## Honest Feedback

Three friction points were clear.

1. **I answered too quickly at one point instead of confirming source truth first.** The user asked a conceptual question about `FieldDescription`, and my first behavior was action-forward when the correct move was explain-first with evidence from shadcn variants. That created avoidable trust friction.

2. **Pattern drift happened during adaptation.** I introduced a reasonable-looking structure that still diverged in subtle ways (`gap` placement, wrapper depth, and description tone). It rendered fine but wasn’t fully aligned with the declared standard. This is exactly the class of drift we said we want to prevent.

3. **Mixed reference sources increased cognitive load.** Pulling from both base and new-york without explicit boundary rules caused back-and-forth edits. I corrected this later by locking Base UI as structural truth, but I should have formalized that sooner.

## Lessons Learned

- For shadcn-based projects, decide **structure source** and **visual source** explicitly before coding.
- For inline copy with links, `<Trans>` is safer than multiple `t` fragments because message context stays intact.
- If user asks “why,” answer with source-backed reasoning first, then implement.
