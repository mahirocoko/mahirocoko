## Session Summary

This session started from token extension intent and then corrected course into brand-system simplification. We moved from exploratory naming (`brand-base`, `brand-accent`, and extra aliases) to a semantic-first model where brand identity is encoded directly through `--primary` and `--accent`, with gradient expressed via start/end stops only. The login page and button variants were iterated several times to improve contrast behavior and remove naming overlap. The main result is a cleaner token contract with fewer aliases, fewer conceptual duplicates, and stronger portability for reuse in future projects.

## Architecture Decisions

- Prefer semantic tokens (`primary`, `accent`) over brand alias layers when alias adds no real abstraction benefit
- Keep gradient as composable stops (`start`/`end`) instead of introducing a combined gradient alias token
- Keep hover behavior for CTA and link interactions away from overly bright accent jumps
- Ensure button variant naming reflects semantics (`primary-gradient`) not temporary design-language labels

## AI Diary

This session felt like a practical lesson in controlled reduction. We started with good intent — making a reusable brand layer — but the first pass overcomplicated the model. I introduced too many names while trying to be future-proof, and the user correctly pushed back each time a naming layer became redundant or confusing. The biggest correction was accepting that `primary` is already the brand anchor in this project, so adding `brand-base` was architectural noise rather than useful abstraction.

I also noticed how quickly color work becomes perceptual instead of purely technical. The screenshot feedback about contrast was crucial: it forced me to stop looking at token names and start looking at actual rendered behavior. We tuned gradient endpoints, hover behavior, and secondary tone in response to visual outcomes, not assumptions. That was the right move.

The strongest part of this session was iterative cleanup with immediate verification: each naming and behavior change was followed by typecheck and doc sync. The weakest part was early drift (including editing Pencil when only reference was needed). Once corrected, the work became cleaner: simplify, validate, document. In the end, the system became both easier to reason about and more reusable because it now depends on fewer concepts.

## Honest Feedback

Three friction points stood out.

1. **I over-layered tokens at first.** I created `brand-*` aliases on top of semantics before proving they were necessary. That violated the simplicity principle the user was clearly aiming for. It took multiple rounds to remove the extra layer and return to semantic-first.

2. **I initially acted on the wrong canvas (Pencil edit) instead of using it as reference.** Even though this was corrected quickly, it introduced avoidable confusion and context switching. The user had explicitly asked for Pencil as reference, not source-of-change.

3. **Variant semantics drifted before stabilizing.** We had `brand`, `brand-outline`, then removed them because `default` already covered one and outline semantics should remain neutral. This indicates I should have performed a sharper overlap check before adding variants.

The positive side is that each friction point converted into a concrete rule that now lives in docs and code. The final state is better than the first proposal because it was pressure-tested by real UI and direct feedback.

## Lessons Learned

- If semantic tokens already carry brand intent, extra alias layers should be avoided until a concrete reuse need appears.
- For gradient tokens, start/end stops are sufficient and clearer than a combined alias token.
- Variant design should start with overlap elimination (`default` vs `brand`) before adding new names.
- Screenshot-driven iteration catches contrast problems faster than class-level reasoning alone.
