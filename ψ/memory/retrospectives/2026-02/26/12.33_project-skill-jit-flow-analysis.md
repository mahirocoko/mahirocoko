## Honest Feedback
Friction point 1: the `/project` skill assumes ghq-centric ownership by default, but real-world work sometimes starts from an existing local clone. I handled it manually via relink, but the workflow still creates extra movement when the user's intent is "use this current path". We should make local-path incubate a first-class branch in the skill.

Friction point 2: documentation drift risk is visible in `jit-flow`. The README presents a broad boilerplate narrative, while the code now represents a much more opinionated Thai-first growth app with journey-centric UX. This mismatch can mislead future contributors and AI agents, causing wrong assumptions in planning.

Friction point 3: many journey/planning screens are rich in UI but still partially static. It looks complete visually, but data wiring depth varies by module. That can create a false sense of done unless we track "UI complete vs data complete" explicitly. I should keep calling this out in analysis so roadmap decisions stay honest.

## Lessons Learned
- Operational correctness starts with path correctness; if workspace indirection is wrong, every downstream action is at risk.
- Repo analysis quality improves when I triangulate across git history, route behavior, service layer, and schema instead of trusting docs alone.
- For product readiness reviews, explicitly label modules as "implemented", "partially wired", or "placeholder" to prevent scope illusion.
